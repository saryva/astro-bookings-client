## 3. Feature Modules

### Module: launches
**Responsibility**: Display available launches and allow navigation to booking.

| Component | Type | Responsibility |
|-----------|------|---------------|
| LaunchListComponent | Page/Container | Fetches data, manages loading/error state, renders list |
| LaunchCardComponent | Presentational | Renders a single launch with its data. Emits `bookNow` event |

### Module: bookings
**Responsibility**: Create, view, and cancel bookings.

| Component | Type | Responsibility |
|-----------|------|---------------|
| BookingFormComponent | Container | Form to select seats, validate, and submit booking |
| BookingListComponent | Page/Container | Fetches bookings for a customer, renders list |
| BookingCardComponent | Presentational | Renders a single booking with status |

### Module: shared
**Responsibility**: Reusable UI components shared across features.

| Component | Responsibility |
|-----------|---------------|
| LoadingSpinnerComponent | Full-screen or inline spinner |
| ErrorMessageComponent | Displays API/generic error message |
| EmptyStateComponent | Empty list feedback with illustration |

## 4. Domain Models

```typescript
// src/app/core/models/launch.model.ts

/** A rocket launch available for customer booking */
export interface Launch {
  id: string;
  rocketName: string;
  launchDate: string;       // ISO 8601
  destination: string;
  pricePerSeat: number;     // USD
  availableSeats: number;
  totalSeats: number;
  status: 'scheduled' | 'launched' | 'cancelled';
}

// src/app/core/models/booking.model.ts

/** A customer booking for a specific launch */
export interface Booking {
  id: string;
  launchId: string;
  launchName: string;
  customerId: string;
  seatsBooked: number;
  totalPrice: number;
  status: 'confirmed' | 'cancelled' | 'pending';
  bookedAt: string;         // ISO 8601
}

/** Payload to create a new booking */
export interface CreateBookingDto {
  launchId: string;
  customerId: string;
  seatsBooked: number;
}
```

## 5. State Management Strategy

**Chosen approach: Angular Signals + Service-level state**

Rationale: The app has simple, local state needs. NgRx would be overkill.
Signals provide reactive state with minimal boilerplate and full Angular 17+ support.

```
API (HttpClient)
    ↓
Service (signal-based state: data, loading, error)
    ↓
Component (reads signals via inject())
    ↓
Template (@if, @for on computed signals)
```

Each service exposes:
```typescript
readonly launches = signal<Launch[]>([]);
readonly loading = signal<boolean>(false);
readonly error = signal<string | null>(null);
```

## 7. API Integration Layer

- Base URL configured in `environment.ts` as `apiBaseUrl`.
- All HTTP calls go through dedicated service classes — never from components.
- `HttpClient` injected via `inject(HttpClient)`.
- Errors caught with `.pipe(catchError(...))` and rethrown as typed errors.

```typescript
// environment.ts
export const environment = {
  production: false,
  apiBaseUrl: 'http://localhost:3000'   // ← change to real API URL
};
```

---

## 8. Error Handling Strategy

| Layer | Strategy |
|-------|---------|
| Service | `catchError` → logs error, rethrows with user-friendly message |
| Component | Sets `error` signal → template shows `<app-error-message>` |
| Global | No global interceptor needed at this stage |
